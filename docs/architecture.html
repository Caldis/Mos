ISSUES问题与改进点
P1-1LogiDeviceSession 巨石：单文件 3,191 行，占子系统 42%
HID++ 协议引擎、4 个隐式状态机（特征发现 5s 超时 / REPROG 顺序查询 / 上报查询超时链 / 接收器枚举 phase 0-2）、定时器簇、特征缓存、合成释放全部纠缠在一个类里。改动风险高度集中，而该区域回归依赖昂贵的真实设备测试。
Mos/Logi/Core/LogiDeviceSession.swift（3,191 行）· 接收器枚举 phase 标志 :101-104, 764-795
按职责拆出 DiscoveryEngine / ReportingQueryEngine / DivertController / ReceiverEnumerator，Session 只保留连接与报文路由。FakeLogiDeviceSession 已证明接缝可行，可逐个状态机搬迁、每步跑 Logi 单测矩阵。
P1-2核心三角循环依赖：ScrollCore ⇄ InputProcessor ⇄ ShortcutExecutor
三个单例互相直呼：滚动核心调输入处理器标记会话，输入处理器调执行器，执行器又回调滚动核心。无法独立实例化测试任何一角，初始化顺序是隐式的，新动作类型会继续加深耦合。
ScrollCore.swift:76-77 → InputProcessor · InputProcessor.swift → ShortcutExecutor · ShortcutExecutor.swift:206 → ScrollCore.handleMosScrollAction
把"Mos 滚动动作"抽成中立端口协议（仿 LogiExternalBridge 的成功经验反向应用）：执行器依赖 ScrollActionPort 而非具体单例，ScrollCore 实现该端口。一次小型协议引入即可断环。
P1-3Options 变更传播缺失 + 全量写放大
任一属性 didSet 都触发全部配置键同步重写（含绑定/应用列表的 JSON 编码）——拖动滑杆期间高频全量落盘；无订阅机制，消费方靠手动 sync 与 isDirty 缓存失效；"readingOptionsLock"实为布尔重入标志而非锁，CVDisplayLink 线程读取配置无同步保证。
Options.swift:68-78（didSet 全保存）· :148-200（全量写）· :58（布尔"锁"）· ScrollPoster.swift:414-418（帧线程读取）
三步走：① 脏键写入 + 短 debounce；② Options 提供变更订阅（closure 列表足矣），替换手动 sync；③ 热路径改读不可变快照（滚动开始时取一次，已有雏形），帧线程不再直读共享结构。
P2-1线程约定的执行力度两极分化
Logi 域用 precondition 把"主线程 only"焊死，而核心域（InputProcessor.activeBindings、ScrollCore 热键布尔、ButtonUtils 缓存）零断言零锁。当前拓扑下确实安全（都在主 RunLoop），但这是隐式约定——未来任何人把回调挪到队列上就会悄无声息地引入竞态。
对照: UsageRegistry.swift（有断言）vs InputProcessor.swift（无断言、无锁的字典变更）
统一 assertMainThread 辅助函数撒到核心域入口（10.13 限制下 @MainActor 不可用，断言是最低成本的固化方式）。
P2-2UI 巨石群：323KB 单体 storyboard + 800 行级视图类
所有窗口挤在一个 Main.storyboard（323KB，合并冲突高发、object id 还是持久化兼容面）；OpenTargetConfigPopover 888 行、ButtonTableCellView 861 行、LogiDebugPanel 2,392 行——视图、状态、业务在同一文件里。
Mos/Base.lproj/Main.storyboard（323KB）· ButtonsView/OpenTargetConfigPopover.swift · ButtonTableCellView.swift
不必大迁移：约定新窗口/新视图一律代码构建（Toast 体系已证明可行），存量仅在大改时顺手拆分。

P3-1权限轮询定时器引用管理松散
启动路径创建的 10s 轮询 Timer 未存引用，sessionDidResign 只能取消 2s 恢复 Timer；反复睡眠/切换用户的极端序列下可能叠加多个轮询器。幂等守卫（isActive 检查）兜住了正确性，但属于侥幸而非设计。

P3-2云端的三项"已知取舍"，保持观察即可

ROADMAP建议的改进顺序
